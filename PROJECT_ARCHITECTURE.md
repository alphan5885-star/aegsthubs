# Project Architecture Overview

## High‑Level Overview
The application is now a **pure Bitcoin (zPub) escrow system** with no external payment gateway dependencies. All payment‑related interactions are handled locally:

1. **Address Generation** – `AddressGenerator` derives a unique, single‑use Bitcoin address from the extended public key (`BTC_XPUB`).
2. **Checkout UI** – `LocalCheckout` shows the generated address, amount, and QR code to the customer.
3. **Escrow Watcher** – `src/lib/escrow.ts` monitors the blockchain (via the BlockCypher API or similar) through a Tor SOCKS5 proxy, validates the transaction and marks the order as **paid**.
4. **Order Lifecycle** – The order record (`orders` table) stores `payment_address`, `payment_status`, and transaction hash. Once confirmed, a Supabase trigger (`trg_order_paid`) updates the order state.

## Payment Flow (Step‑by‑Step)
1. **Order Creation** – A user creates an order; the backend records it in `orders` with status `pending`.
2. **Address Generation** – `LocalCheckout` calls `AddressGenerator.generate()` which:
   - Uses the master `BTC_XPUB` from the environment.
   - Derives a child index based on the order ID (or a monotonic counter).
   - Returns a fresh Bitcoin address and its accompanying scriptPubKey.
3. **Display to User** – The address and QR code are rendered. The UI also shows a helpful tooltip explaining what a zPub is.
4. **Escrow Monitoring** – `src/lib/escrow.ts` runs every 30 seconds (configurable) and:
   - Queries the blockchain for unspent outputs at the generated address.
   - Verifies the received amount meets or exceeds the order total.
   - Calls the Supabase RPC `rpc('trg_order_paid')` to transition the order to `paid`.
5. **Order Fulfilment** – Once the order is marked `paid`, the rest of the system (shipping, notifications, etc.) proceeds as before.

## Security Considerations
- **No Private Keys** – Only the read‑only xpub/zpub is stored; private keys never leave the secure hardware wallet used by the merchant.
- **Tor‑Based Queries** – All blockchain look‑ups are performed over Tor to hide the merchant’s IP.
- **Transaction Confirmation** – The escrow service waits for a configurable number of confirmations (default 3) before marking an order as paid.
- **Environmental Isolation** – All secrets are kept in `.env` and never committed to source control.

## Future Extensibility
The architecture is deliberately modular. Adding support for additional cryptocurrencies (e.g., LTC, XMR) would involve:
1. Extending `AddressGenerator` with the appropriate derivation scheme.
2. Updating `LocalCheckout` UI to display the new address/QR code.
3. Enhancing `escrow.ts` to monitor the new blockchain network.
All other components remain unchanged.
